# Contributing

The corpus is regenerated by a human running the generator and opening
a PR. There is no automation that pushes commits directly.

## Setup

You need:

- [Bun](https://bun.sh/) (the generator + validator are TypeScript run
  via Bun, no `tsc` step).
- An `EXEMEM_DEV_API_KEY` — mint one at
  <https://www.exemem.com/developer>. The generator uses it to fetch
  `/v1/snapshot` from the production schema_service.
- An `ANTHROPIC_API_KEY` for the LLM that drafts the rows.

```bash
cd tools
bun install
```

## Regenerating samples

```bash
# Targeted regen — replace specific schemas
EXEMEM_DEV_API_KEY=em_... ANTHROPIC_API_KEY=sk-... \
  bun generate.ts --schemas Message,Recipe,Movie

# Full regen — every schema in the snapshot
EXEMEM_DEV_API_KEY=em_... ANTHROPIC_API_KEY=sk-... \
  bun generate.ts --all
```

Row count per schema is `10 + field_count`, capped at 100. The
generator writes `samples/<descriptive_name>.json` (pretty-printed
JSON array) and updates `manifest.json` in place.

## Validating

```bash
bun tools/validate.ts samples/
```

The validator pulls the current snapshot from schema_service and
asserts every row in every file conforms to the schema's declared
`field_types`. CI runs the same command on every PR.

If a schema has drifted between the snapshot the file was generated
against and the current snapshot, the validator surfaces the
discrepancy — regenerate just that schema with
`bun generate.ts --schemas <name>`.

## Filing a PR

1. Branch from `main`.
2. Run the generator for the schemas you want to update.
3. `bun tools/validate.ts samples/` — must pass.
4. Commit `samples/*.json` and the updated `manifest.json`.
5. Open a PR. CI will rerun validation.

## What goes in v1 vs. follow-ups

v1 covers ~20 well-known canonical schemas (Message, Book, Recipe,
Article, Movie, Review, etc.). Broader coverage of the ~956-schema
registry is incremental — each "add N more schemas" change is its own
PR. Don't try to regenerate everything in one go unless you have a
specific reason.

## Anonymization

The generator's prompt instructs the LLM to avoid real names, emails,
phone numbers, and addresses. The validator can only enforce *shape*,
not semantic content — if you notice a sample that looks like real data
(or anything else that should not be public), open an issue.
